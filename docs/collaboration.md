### 2. 开发工作流

#### Step 1: 创建功能分支

```bash
git checkout dev
git pull origin dev
git checkout -b feature/新功能名称
```

#### Step 2: 本地开发（元宝）

- 在WorkBuddy中编写代码
- 使用对应的Skill（如 `quant-math-engine`）
- 本地测试通过

#### Step 3: 云端协同（豆宝）

- 豆宝在Coze云端接收任务
- 开发在线API服务

#### Step 4: 代码审查（CODEX + 包青天）

- CODEX自动审计代码质量
- 包青天检查合规性（尤其是金融数据使用）
- 通过后合并到 `dev` 分支

#### Step 5: 测试验证（云宝）

- 云宝在沙箱环境完整测试
- 验证双轨制预测准确率
- 自动复盘迭代（如准确率<95%）

#### Step 6: 部署上线

- 本地：元宝更新WorkBuddy Skills
- 云端：豆宝部署Coze API服务
- 通知老大：任务完成

## 🌿 分支策略

```
main (稳定版)
 └── dev (开发版)
      ├── feature/双轨融合引擎
      ├── feature/回测系统
      ├── feature/在线API
      └── bugfix/修复L4b权重
```

### 分支说明

- **main**：生产稳定版，只允许Hotfix
- **dev**：开发主分支，所有功能合并到此
- **feature/***：功能开发分支
- **bugfix/***：Bug修复分支
- **hotfix/***：紧急生产修复

## 📝 提交规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/)：

### 提交格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type类型

| Type | 说明 | 示例 |
|------|------|------|
| **feat** | 新功能 | `feat(fusion): 添加双轨融合引擎` |
| **fix** | Bug修复 | `fix(L4b): 修正调仓权重+15%→+25%` |
| **docs** | 文档更新 | `docs(README): 更新协作指南` |
| **style** | 代码格式 | `style(fusion): 修复PEP8警告` |
| **refactor** | 重构 | `refactor(math): 优化因子计算性能` |
| **test** | 测试 | `test(api): 添加融合引擎单元测试` |
| **chore** | 构建/工具 | `chore(deps): 升级Flask到2.3.0` |

### 示例

```bash
git commit -m "feat(fusion): 实现Track A/B双轨融合

- 添加fusion_engine.py
- 支持L0流动性调整
- 支持L4b调仓效应调整

Closes #12"
```

## ✅ 代码审查规范

### CODEX自动审计（P0优先级）

所有PR必须通过的自动检查：

1. **代码质量**：PEP8、复杂度<10
2. **安全审计**：无硬编码密钥、无SQL注入风险
3. **性能检查**：无死循环、无内存泄漏
4. **测试覆盖**：新功能必须有单元测试

### 包青天合规检查（金融数据专用）

涉及金融数据的代码必须检查：

1. **数据源合规**：仅使用免费/已授权数据（AKShare、Tushare）
2. **用户隐私**：不收集/存储用户个人信息
3. **预测免责**：所有预测结果必须标注"仅供参考，不构成投资建议"
4. **风险控制**：所有交易策略必须包含止损逻辑

### Review流程

1. 提交PR到 `dev` 分支
2. CODEX自动审计（<5分钟）
3. 包青天合规检查（如有金融数据）
4. 人工Review（至少1人）
5. 所有检查通过后合并
