# 架构说明 (Architecture)

> 九章量化局 v2.0 系统详细设计文档

---

## 🎯 设计理念

**双轨制验证**：AI定性（理解） + 数学定量（精确） = 高准确率预测

**自动迭代**：预测失败 → 自动复盘 → 更新因子 → 重新测试

**高度协同**：9个Agent按角色分工，飞龙在天协议统一调度

## 📊 核心模块详解

### 1. Track A（AI定性分析）

#### 职责

通过大语言模型（Claude/GPT/GLM）理解市场本质，捕捉数学因子无法捕捉的信号：

- **市场情绪**：恐慌/贪婪指数、舆情分析
- **政策影响**：央行货币政策、产业政策、监管变化
- **行业趋势**：技术革新、竞争格局、供需变化
- **事件驱动**：财报、并购、高管变动

#### 输入

- 历史价格数据（通过TDX Connector）
- 新闻资讯（通过wenda_news_query）
- 宏观数据（通过wenda_macro_query）
- 政策文件（通过IMA知识库）

#### 输出

- 趋势判断（看多/看空/震荡）
- 关键催化剂（近期可能推动价格的因素）
- 风险提示（黑名单ETF、系统性风险）

#### 实现

- **Skill**: `quant-analyst`, `quant-vibe-only`
- **工具**: TDX Connector, IMA MCP, WebSearch
- **模型**: WorkBuddy支持的任何LLM

---

### 2. Track B（数学因子打分）

#### 职责

通过6大类、20+子因子，对ETF进行客观量化评分：

| 因子类别 | 权重 | 说明 |
|---------|------|------|
| **动量因子** | 35% | 5/10/20日均线、MACD、RSI |
| **资金流因子** | 25% | 北向资金、主力资金净流入 |
| **均值回归因子** | 15% | 布林带位置、乖离率 |
| **波动率因子** | 10% | ATR、历史波动率 |
| **微观结构因子** | 10% | 指数调仓冲击、被动资金流入 |
| **市场状态调整** | 5% | 牛市/熊市/震荡市调整 |

#### 输入

- ETF历史K线数据（通过 `tdx_kline`）
- 实时行情数据（通过 `tdx_quotes`）
- 资金流数据（通过 `tdx_wenda_query`）
- 指数成分调整公告（通过 `lark-doc` 或WebSearch）

#### 输出

- 综合得分（0-100分）
- 各因子得分明细
- 买入/卖出信号强度

#### 实现

- **Skill**: `quant-math-engine`
- **核心脚本**: `factor_calculator.py`
- **因子目录**: `references/factor_catalog.md`

#### 公式示例

**动量因子得分**：

```
momentum_score = 
  0. 4 * (close - ma5) / ma5 +
  0. 3 * (close - ma10) / ma10 +
  0. 2 * (close - ma20) / ma20 +
  0. 1 * MACD_histogram
```

**资金流因子得分**：

```
capital_flow_score =
  0. 5 * northbound_flow_5d +
  0. 3 * main_force_flow_5d +
  0. 2 * volume_ratio
```

> 详见 `skills/quant-math-engine/references/factor_catalog.md`

### 3. 融合引擎 (fusion_engine.py)

#### 职责

将Track A和Track B的输出融合，生成最终预测报告。

#### 融合逻辑

```
最终得分 = Track B得分 
          + L0流动性调整 (宽松+10%, 紧张-20%)
          + L4b调仓效应调整 (冲击系数>3时+25%)
          + Track A情绪调整 (看多+5%, 看空-5%)
```

#### 一致性检查

- 如果Track A和Track B方向一致（都看多/都看空）→ 置信度+20%
- 如果方向分歧 → 标记为"需人工复核"，降低仓位

#### 输出格式

```json
{
  "etf_code": "515880",
  "etf_name": "通信ETF",
  "track_b_score": 78.5,
  "liquidity_adj": +2.3,
  "rebalance_adj": +4.1,
  "fused_score": 84.9,
  "track_a_outlook": "看多",
  "consistency": "一致",
  "confidence": "高",
  "predicted_return_3d": "+5.2%",
  "stop_loss": "-3.0%"
}
```

#### 实现

- **脚本**: `dashboard/fusion_engine.py`
- **更新频率**: 每个交易日 15:30 自动运行
- **输出**: `fusion_report.json`

### 4. 系统性因子监测 (systematic-factor-monitor)

#### L0：流动性检测

**目标**：捕捉央行货币政策变化对市场的整体影响。

**监测指标**：

| 指标 | 数据来源 | 调整规则 |
|------|---------|---------|
| 央行OMO操作 | 365与时俱进（IMA知识库） | 净投放>1000亿 → +10%权重 |
| MLF利率 | 365与时俱进 | 降息 → +15%权重 |
| DR007 | 同花顺/Wind | 低于政策利率 → +5%权重 |
| 北向资金 | TDX Connector | 连续3日净流入>50亿 → +10% |

**实现**：

- **Skill**: `systematic-factor-monitor`
- **更新频率**: 每个交易日 09:00（开盘前）

#### L4b：微观结构冲击检测

**目标**：识别指数调仓、ETF申购赎回等微观结构变化对ETF价格的冲击。

**监测事件**：

| 事件 | 冲击系数 | 调整规则 |
|------|---------|---------|
| 沪深300调仓 | 0.8 | 新纳入ETF +1.5%权重 |
| 中证500调仓 | 0.6 | 新纳入ETF +1.0%权重 |
| ETF大额申购 | 0.4 | 单日申购>10亿 → +0.5% |
| 指数成分调整 | 0.9 | 权重变化>5% → +2.0% |

**计算公式**：

```
rebalance_adj = track_b_score * 冲击系数 * 0.25
```

> **v1.1修正**：冲击系数>3时，调整权重从+15%提升到+25%

**实现**：

- **数据源**: 指数公司公告（通过WebSearch或lark-doc）
- **更新频率**: 调仓日前5天开始监测

### 5. 九章Agent协同系统 (quant-agent-orchestra)

#### 角色定义

| ID | 角色名称 | 核心职责 | 使用Skill |
|----|----------|----------|-----------|
| 01 | **大A** | A股哨兵，Track A分析 | quant-analyst |
| 02 | **老美** | 美股侦察兵，Track A分析 | quant-analyst |
| 03 | **军机处** | 宏观情报官，L0/L1监测 | systematic-factor-monitor |
| 04 | **量化1哥** | 数学模型，Track B计算 | quant-math-engine |
| 05 | **包青天** | 合规审计官，风险检查 | compliance-auditor |
| 06 | **史官** | 记录官，自动复盘 | quant-recap-engine |
| 07 | **执行者** | 交易执行官（模拟盘） | tdx-connector |
| 08 | **风控** | 止损止盈管理 | finance-risk-manager |
| 09 | **协调官** | 九章协调官，冲突解决 | agent-orchestra |

#### 协同协议

**飞龙在天协议**（六步）：

1. **同频邮件**：豆宝+天宝同步最新情报
2. **读KB**：读取老大画像KB（3天内变更）
3. **更新状态**：各Agent更新自身状态
4. **汇总报告**：协调官汇总各Agent报告
5. **唤醒豆宝**：通知云端Agent开始工作
6. **通知老大**：通过QQ邮箱发送结果

**冲突解决机制**：

- **Track A vs Track B分歧**：协调官组织辩论，史官记录
- **因子权重争议**：回测数据说话，取历史表现更优者
- **合规风险**：包青天一票否决

#### 实现

- **Skill**: `quant-agent-orchestra`
- **触发词**: "飞龙在天"、"开始协同预测"

- **本地**: `http://localhost:7860`

#### 自动刷新

- 前端每15秒自动刷新数据
- 后端每个交易日 15:30 自动更新 `fusion_report.json`

### 7. 自动复盘迭代系统 (quant-recap-engine)

#### 触发条件

- 预测准确率 < 目标（Phase 1: 70%, Phase 2: 85%, Phase 3: 95%）
- 单个ETF预测错误（实际涨跌与预测相反）
- 用户手动触发（"开始复盘"）

#### R1-R6流程

| 步骤 | 名称 | 说明 |
|------|------|------|
| **R1** | 收集失败案例 | 从 `fusion_report.json` 提取错误预测 |
| **R2** | 分析根本原因 | 是因子失效？还是数据质量问题？ |
| **R3** | 提出改进方案 | 调整因子权重？增加新因子？ |
| **R4** | 更新因子权重 | 修改 `factor_calculator.py` |
| **R5** | 重新测试 | 用最近3个月数据回测 |
| **R6** | 归档经验教训 | 更新 `quant-recap-engine/SKILL.md` |

#### 实现

- **Skill**: `quant-recap-engine`
- **自动化**: 通过 `automation_update` 设置定时任务（每周一 08:00）

## 🔄 数据流

### 预测流程（每个交易日 15:30）

```
1. 军机处 → 读取L0流动性数据
2. 大A/老美 → Track A分析（AI定性）
3. 量化1哥 → Track B计算（数学因子打分）
4. 融合引擎 → 融合Track A/B，输出fusion_report.json
5. 包青天 → 合规检查
6. 实时看板 → 更新数据，刷新图表
7. 史官 → 记录到日志（~/WorkBuddy/.workbuddy/memory/YYYY-MM-DD.md）
```

### 复盘流程（每周一 08:00）

```
1. 史官 → 读取上周预测记录
2. 计算准确率 → 对比预测vs实际
3. 如果准确率 < 目标 → 触发R1-R6
4. 更新因子权重 → 提交到Git
5. 通知老大 → 发送QQ邮件
```

## 📈 性能优化

### 当前瓶颈

1. **TDX数据获取慢**（~3秒/次）
2. **因子计算未并行**（6大类因子串行计算）
3. **前端图表渲染卡顿**（>50个ETF时）

### 优化方案

| 瓶颈 | 方案 | 优先级 |
|------|------|--------|
| TDX慢 | 本地缓存（SQLite） | P0 |
| 串行计算 | 多线程/多进程 | P1 |
| 渲染卡顿 | 虚拟滚动（前端） | P2 |

## 🔒 安全设计

### 数据安全

- ✅ 不收集用户个人信息
- ✅ 所有预测结果标注"仅供参考"
- ✅ 金融数据仅使用免费/已授权源

- ✅ GitHub仓库公开（开源）
- ✅ 但敏感配置不提交（如API Key）
