# 自动复盘迭代机制 —— 详细流程

## 触发条件

**每次预测结束后**，自动执行以下检查：

```
触发检查点：
1. A股收盘后（15:00） → 触发复盘检查
2. 美股收盘后（次日05:00 EST） → 触发复盘检查
3. 用户主动询问"准确度"时 → 立即执行复盘
```

---

## 复盘迭代流程（R1-R6）

### R1: 数据拉取

**目标**：获取预测对应的实际市场数据。

**执行步骤**：

```
1. 确定预测时点：
   - 从对话历史/记忆文件中提取"上次预测内容"
   - 提取：预测日期、预测指数涨跌幅、推荐ETF及预测涨跌幅

2. 拉取实际数据：
   a. 使用TDX工具拉取指数实际收盘涨跌幅
      代码：
      - 上证指数：000001（setcode=1）
      - 深证成指：399001（setcode=0）
      - 创业板指：399006（setcode=0）
      - 科创50：000688（setcode=1）

   b. 使用TDX工具拉取推荐ETF实际涨跌幅
      代码：从预测记录中提取ETF代码

   c. 若TDX无法获取（如美股ETF） → 使用WebSearch搜索"[ETF代码] 收盘 2026年X月X日"

3. 格式化数据（与预测时相同格式）：
   输出YAML格式：
   ```yaml
   prediction_date: "2026-06-15"
   actual_data:
     indices:
       - name: "上证指数"
         predicted: "+1.0%"
         actual: "+1.61%"
         error: "0.61%"
       - name: "创业板指"
         predicted: "+1.0%"
         actual: "+5.30%"
         error: "4.30%"
     etfs:
       - code: "515880"
         name: "通信ETF"
         predicted_rank: 6
         actual_return: "+6.63%"
         actual_rank: 1
         rank_error: "5位（严重失误）"
   ```
```

### R2: 准确度计算

**目标**：量化预测准确度，识别未达标项。

**准确度指标定义**：

| 指标 | 计算方法 | 目标值 | 计算公式 |
|------|---------|--------|---------|
| 方向准确度 | 预测涨/跌 vs 实际涨/跌，方向一致则正确 | **>95%** | (正确方向数 / 总预测数) × 100% |
| 幅度准确度 | 预测涨跌幅与实际涨跌幅误差 < 1% 为准确 | **>80%** | (误差<1%的数量 / 总预测数) × 100% |
| ETF排序准确度 | 预测排名 vs 实际涨幅排名 | **Spearman系数 >0.85** | 1 - (6 × Σd²) / (N³ - N) |
| ETF涨幅准确度 | 推荐ETF实际涨幅在预测涨幅区间内 | **>70%** | (在区间内数量 / 总ETF数) × 100% |

**执行步骤**：

```
1. 计算方向准确度：
   a. 对每个指数预测，提取"预测方向"（涨/跌/震荡）和"实际方向"
   b. 若预测"涨"且实际涨 → 正确
      预测"跌"且实际跌 → 正确
      预测"震荡"且实际涨跌幅在±1%内 → 正确
      其他 → 错误
   c. 计算：方向准确度 = (正确数 / 总指数数) × 100%

2. 计算幅度准确度：
   a. 对每个指数，计算 |预测涨跌幅| - |实际涨跌幅||
   b. 若误差 < 1% → 准确
   c. 计算：幅度准确度 = (准确数 / 总指数数) × 100%

3. 计算ETF排序准确度：
   a. 提取预测ETF排名（按推荐顺序）和实际ETF排名（按实际涨幅）
   b. 计算Spearman等级相关系数：
      formula: ρ = 1 - (6Σdᵢ²) / (N³ - N)
      其中 dᵢ = 预测排名 - 实际排名，N = ETF数量
   c. 判定：
      - ρ > 0.85 → 优秀
      - ρ 0.6-0.85 → 良好
      - ρ < 0.6 → 需改进

4. 输出准确度报告：
   ```yaml
   accuracy_report:
     direction_accuracy: 100%  # 3/3正确
     magnitude_accuracy: 33%   # 1/3误差<1%
     etf_rank_accuracy: 0.25    # Spearman系数（很低）
     etf_return_accuracy: 17%    # 1/6在区间内
     overall_pass: false          # 未通过（幅度准确度<80%）
   ```
```

### R3: 归因分析

**目标**：识别预测失误的根本原因。

**归因分类框架**：

| 归因类型 | 描述 | 升级动作 |
|---------|------|---------|
| 因子漏检 | 未检测到的关键因子（如"指数调仓"） | 扩展检测清单 |
| 因子误判 | 检测到但判定的方向/幅度错误 | 调整权重/阈值 |
| 因子缺失 | 完全未纳入考虑的因子类型 | 新增检测维度 |
| 排序算法缺陷 | ETF排序逻辑不合理 | 重写排序算法 |
| 数据源误差 | 使用的数据本身不准确 | 更换数据源/交叉验证 |
| 黑天鹅事件 | 未预期的突发事件（如政策急转弯） | 加入"尾部风险"检测 |

**执行步骤**：

```
1. 对每个预测失误项，执行归因分析：

   示例：创业板指预测+1.0%，实际+5.30%，误差4.30%
   - Step 1: 检查L0流动性检测是否执行
     → 发现：执行了，但"央行6000亿逆回购"因子权重低估
   - Step 2: 检查L4b微观结构检测是否执行
     → 发现：未执行！"科创50调仓"因子完全漏检
   - Step 3: 判定归因
     → 主要归因：因子漏检（L4b）
     → 次要归因：因子低估（L0流动性权重）

2. 输出归因报告：
   ```yaml
   attribution_analysis:
     - prediction: "创业板指 +1.0%"
       actual: "+5.30%"
       error: "4.30%"
       root_cause: "因子漏检"
       details:
         - missed_factor: "科创50调仓效应（冲击系数5.0）"
           severity: "🔴严重"
         - underestimated_factor: "央行6000亿逆回购"
           severity: "🟡中等"
       upgrade_actions:
         - action: "将L4b检测提前至L0之后立即执行"
           priority: "P0"
         - action: "L0流动性宽松时，高Beta品种权重+20% → +30%"
           priority: "P1"
   ```

3. 识别"新因子"：
   - 检查是否有在本次预测中未纳入，但实际产生了影响的因子
   - 若有 → 加入"检测信号清单"（SKILL.md中的表格）
```

### R4: 升级检测逻辑

**目标**：将归因分析的结论转化为系统升级动作。

**升级动作清单**：

| 升级类型 | 具体操作 | 影响文件 |
|---------|---------|---------|
| 扩展检测清单 | 在SKILL.md的"检测信号清单"表格中新增一行 | SKILL.md |
| 调整判定阈值 | 修改"流动性状态判定标准"中的阈值 | references/liquidity_detection.md |
| 调整权重 | 修改"因子权重调整规则"表 | SKILL.md |
| 重写排序算法 | 修改"因子权重输出"部分的逻辑 | SKILL.md |
| 新增数据源 | 在"数据源清单"中新增数据源 | references/liquidity_detection.md |
| 新增归因类型 | 在"归因分类框架"中新增类型 | references/recap_mechanism.md |

**执行步骤**：

```
1. 根据归因分析报告，生成升级动作清单：

   示例：
   归因：L4b因子漏检（科创50调仓）
   → 升级动作：
      a. 将"指数事件检测"从"可选步骤"提升为"必须步骤"
      b. 在SKILL.md的"调用时机"中明确："每次分析必须先执行Step 2"
      c. 将"调仓效应"在ETF排序中的权重从+15%提升至+25%

2. 执行升级动作：
   a. 使用Edit工具修改SKILL.md
   b. 使用Edit工具修改references/文件
   c. 验证修改：重新读取文件，确认修改已生效

3. 记录升级内容：
   - 写入 ~/.workbuddy/memory/recap-log.md
   - 格式：升级日期、升级内容、影响文件、预期效果
```

### R5: 调整因子权重

**目标**：根据历史准确度，动态调整各因子在排序中的权重。

**权重调整算法**：

```
初始权重（第一次预测前设定）：
  L0流动性宽松：+20%（高Beta）/ -10%（低Beta）
  L4b调仓效应：+15%
  L1赛道结构：±0%（基准）
  L4供需缺口：±0%
  L5催化剂：±0%

动态调整规则（每次复盘后执行）：
  IF L0因子预测准确度 > 95% → 权重 +5%
  IF L0因子预测准确度 < 80% → 权重 -5%
  IF L4b因子预测准确度 > 95% → 权重 +5%
  IF L4b因子预测准确度 < 80% → 权重 -5%
  
  （其他因子同理）

权重边界：
  - 最大权重：+50%
  - 最小权重：-30%
  - 防止权重过度集中导致其他因子失效
```

**执行步骤**：

```
1. 计算各因子的历史预测准确度（过去5次预测）
2. 根据动态调整规则，计算新权重
3. 更新SKILL.md中的"因子权重调整规则"表
4. 输出权重变化日志：
   ```yaml
   weight_adjustment_log:
     - date: "2026-06-16"
       factor: "L4b调仓效应"
       old_weight: "+15%"
       new_weight: "+25%"
       reason: "漏检导致严重失误，提升优先级"
   ```
```

## 自动化实现方案

### 方案A：手动触发（当前）

在每次预测后，用户询问"准确度如何"或"复盘一下"时，手动执行R1-R6。

**优点**：灵活，人工可以控制复盘时机  
**缺点**：可能忘记复盘，导致系统无法持续进化

---

### 方案B：自动提醒（推荐）

设置一个"复盘提醒"机制：

```
实现方式：
1. 在预测完成后，创建一个"复盘任务"提醒
2. 提醒时间：A股收盘后（15:30）/ 美股收盘后（次日05:30 EST）
3. 提醒内容："上次预测已可验证，是否执行复盘？"
4. 用户确认后，自动执行R1-R6
```

**使用工具**：`automation_update` 创建一次性提醒（scheduleType="once"）

---

### 方案C：全自动（终极目标）

训练一个"复盘Agent"，在预测完成后自动：
1. 等待市场收盘
2. 自动拉取实际数据
3. 自动执行R1-R6
4. 自动更新Skill文件
5. 输出复盘报告给用户

**当前限制**：需要API调用权限和文件系统写入权限，可能超出当前工具范围。

```yaml

- prediction_date: 2026-06-15
- overall_accuracy: 33%（幅度准确度）
- root_cause: "L4b因子漏检 + L0权重低估"

    - "L4b检测提升为必须步骤"

    - "高Beta权重在流动性宽松时+30%（原+20%）"

## [下次复盘]

## 注意事项

1. **复盘必须在预测后可验证时执行** —— 若预测的是"下周走势"，则需等待下周收盘后才能复盘
2. **升级动作必须写入文件** —— 不能只输出报告而不修改Skill，否则系统无法真正"进化"
3. **权重调整必须设定边界** —— 防止某个因子权重过高导致其他因子失效
4. **复盘报告必须简洁** —— 重点突出"误差根因"和"升级动作"，不要堆砌数据
5. **目标明确** —— 每次复盘都以"准确度>95%"为目标，未达标则必须迭代
